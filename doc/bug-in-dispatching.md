# Bug: Master overload dispatcher can pick the wrong overload when a
# defaulted/optional trailing parameter ties in declared arity with a
# shorter, fully-required overload

**Status: fixed in Generator Version 49.** See
`doc/generator-design-notes.md`'s "Fixing Master Wrapper Dispatch Ordering and
Non-Primitive Type Checks (Version 49)" section for the fix actually applied
(`mandatory-param-count`/`master-wrapper-more-specific-p` for root cause 1;
DotCL 0.1.18's `dotnet:is-instance-of` for root cause 2 — a simpler single-call
alternative to the `dotnet:resolve-type` + manual `IsInstanceOfType` sketch
this doc originally proposed below). The rest of this document is kept as the
original bug report/analysis for historical reference.

## Symptom

Discovered downstream in `dotcl-dungeonslime` (a consumer of this
generator's output). Calling the generated master wrapper for
`Microsoft.Xna.Framework.Media.MediaPlayer.Play` with a single argument —
`(media-player:play song)`, i.e. the plain `Play(Song)` overload, no
start-position — throws a .NET `MissingMethodException` at runtime instead
of invoking `Play(Song)`:

```
Warning: play-song failed: Method 'Microsoft.Xna.Framework.Media.MediaPlayer.Play' not found.
[.NET MissingMethodException]
   at System.RuntimeType.InvokeMember(...)
   ...
```

The generated dispatcher silently calls `MediaPlayer.Play(song, 0)` instead
of `MediaPlayer.Play(song)`. There is no real overload matching `(Song,
Int32)`, so .NET's reflection binder throws.

This is **not environment-specific** (not a REPL-vs-standalone-build
difference, not a threading issue) — it is a pure Lisp logic bug in the
generated dispatch `cond`, and reproduces identically every time the affected
call shape is used, in any host process.

## The generated code (real example)

File: `cspackages/microsoft-xna-framework-media-media-player.lisp` in the
downstream repo, generated by this tool at Version 48:

```lisp
(cl:defun play (song cl:&optional (start-position cl:nil supplied-start-position))
  "Master wrapper for Microsoft.Xna.Framework.Media.MediaPlayer.Play overloads. Dispatches at runtime.

Play(Song) -> Void

Play(Song, TimeSpan]) -> Void

Play(SongCollection, Int32 = 0) -> Void
"
  (cl:cond
    ((cl:and (cl:or (cl:null song) (dotnet:object-type song)) supplied-start-position (cl:or (cl:null start-position) (dotnet:object-type start-position)))
     (dotnet:static <type-str> "Play" song start-position))
    ((cl:and (cl:or (cl:null song) (dotnet:object-type song)) (cl:or (cl:not supplied-start-position) (cl:numberp start-position)))
     (dotnet:static <type-str> "Play" song (cl:if supplied-start-position start-position 0)))
    ((cl:and (cl:or (cl:null song) (dotnet:object-type song)) (cl:not supplied-start-position))
     (dotnet:static <type-str> "Play" song))
    (cl:t (cl:error 'csharp-assembly-utils:csharp-overload-not-found
                    :package-name "MICROSOFT-XNA-FRAMEWORK-MEDIA-MEDIA-PLAYER"
                    :class-name <type-str>
                    :method-name "Play"
                    :supplied-args (cl:append (cl:list :song song) (cl:when supplied-start-position (cl:list :start-position start-position)))))))
```

Three real .NET overloads are being dispatched here:

1. `Play(Song)` — 1 required parameter.
2. `Play(Song, TimeSpan)` — 2 required parameters.
3. `Play(SongCollection, Int32 = 0)` — 1 required + 1 parameter with a usable
   default.

Calling `(play song)` with no second argument sets `supplied-start-position`
to `NIL`. Clause evaluation order:

- Clause 1 (`Play(Song,TimeSpan)`): requires `supplied-start-position` to be
  true. Fails, skipped.
- Clause 2 (`Play(SongCollection,Int32=0)`): its guard is `(or (not
  supplied-start-position) (numberp start-position))`. Since
  `supplied-start-position` is `NIL`, `(not supplied-start-position)` is
  true, so **this clause matches unconditionally whenever the caller omits
  the second argument** — regardless of what `song` actually is. It fires,
  calling `Play(song, 0)`.
- Clause 3 (`Play(Song)`, the one actually wanted): its guard `(and (...)
  (not supplied-start-position))` is a **strict subset** of clause 2's guard.
  It can never be reached — dead code.

Result: every plain `(play song)` call is silently miscompiled into
`Play(song, 0)`, which matches no real overload, so .NET throws
`MissingMethodException`.

## Root causes (two, compounding)

### 1. `generate-master-wrapper`'s specificity sort only counts raw
   declared-parameter count, not *effective minimum required* arity

`apg-overload-dispatch.lisp`, `generate-master-wrapper`, around line 500:

```lisp
;; Sort methods by parameter count descending for specificity
(cl:let ((sorted-methods (cl:sort (cl:copy-list methods) #'> :key (cl:lambda (m) (cl:length (cl:getf m :parameters))))))
  (cl:dolist (cm sorted-methods)
    ...))
```

This assumes "more declared parameters = more specific = should be tried
first." That's true when every parameter is mandatory, but breaks down for
an overload with a *usable default* on its trailing parameter: such an
overload is only nominally "2 parameters" but its **effective minimum
arity is 1**, tying it with a genuinely 1-parameter overload
(`Play(Song)`). The sort key doesn't know about `usable-default-p` at all,
so `Play(SongCollection, Int32 = 0)` (raw count 2) is ranked strictly ahead
of `Play(Song)` (raw count 1) even though, for a 1-argument call, they are
equally applicable candidates. Whichever of the two tied-effective-arity
overloads sorts first wins by construction, and the plain, no-default
overload always sorts last among ties since it has the smaller raw count —
making it permanently unreachable whenever a same-prefix overload with a
defaulted trailing parameter also exists.

**Fix direction**: sort by effective minimum required arity (raw parameter
count *minus* the number of trailing parameters with `usable-default-p`)
descending, not raw parameter count. Overloads with defaults should sort
*after* same-arity overloads with no defaults at that tied effective arity,
since the no-default overload is the more specific (exact) match for a call
that omits the trailing argument entirely.

### 2. `format-param-type-check`'s fallback branch for non-primitive types
   cannot discriminate between two different .NET reference/struct types

`apg-overload-dispatch.lisp`, lines 150–160:

```lisp
(defun format-param-type-check (param-type arg-str)
  "Returns a Lisp type checking expression string for the given C# parameter type."
  (cond
    ((member param-type '("System.Double" "System.Single" "System.Int32" "System.Int64" "System.Int16" "System.Byte" "System.Decimal") :test #'string=)
     (format nil "(cl:numberp ~A)" arg-str))
    ((string= param-type "System.Boolean")
     (format nil "(cl:typep ~A 'cl:boolean)" arg-str))
    ((string= param-type "System.String")
     (format nil "(cl:stringp ~A)" arg-str))
    (t
     (format nil "(cl:or (cl:null ~A) (dotnet:object-type ~A))" arg-str arg-str))))
```

For any parameter type that isn't one of the numeric primitives, `Boolean`,
or `String` — i.e. every reference type (`Song`, `SongCollection`,
`Object`, ...) and every other struct (`TimeSpan`, ...) — the generated
check degrades to `(or (null arg) (dotnet:object-type arg))`, which only
asks "is this `nil`, or is this *some* wrapped .NET object at all?" It does
not check that `arg` is actually an instance of the *specific* expected
type. Two overloads that differ only by a non-primitive parameter type at
the same position (e.g. `Song` vs. `SongCollection`) are therefore
indistinguishable to the dispatcher: both produce the exact same guard
expression, `(or (null song) (dotnet:object-type song))`, so ordering alone
decides which one fires. This is why fixing only the sort-order bug (#1)
is insufficient in general — it would resolve *this specific* `MediaPlayer`
case (since `Play(Song)`'s own guard would then be tried before
`Play(SongCollection, Int32=0)`'s), but leaves the dispatcher unable to
correctly route a call that actually supplies a `SongCollection`, or any
other pair of overloads sharing a position with two different non-primitive
types.

**Fix direction**: use a real per-instance type check instead of the
generic "is some object" fallback. Per
`apg-naming.lisp`'s Version 44 changelog entry (this file, search
"class-of/typep on any wrapped .NET object"), DotCL already registers
wrapped .NET types as CLOS classes lazily on first use, and `class-of`/
`typep` on a wrapped .NET object works directly. That suggests
`format-param-type-check`'s fallback branch could emit something like:

```lisp
(format nil "(cl:or (cl:null ~A) (cl:typep ~A '~A))" arg-str arg-str (clos-class-designator-for param-type))
```

or, if CLOS-class dispatch is judged too fragile here (simple-name
collisions across namespaces are a known sharp edge — see the same
changelog entry), an explicit runtime `Type.IsInstanceOfType`-style check
via `dotnet:object-type` compared against `dotnet:resolve-type` for the
specific `param-type`, e.g. something equivalent to:

```lisp
(format nil "(cl:or (cl:null ~A) (cl:and (dotnet:object-type ~A) (dotnet:invoke (dotnet:resolve-type ~S) \"IsInstanceOfType\" ~A)))"
        arg-str arg-str param-type arg-str)
```

(exact DotCL API to use for an `is`/`IsInstanceOfType`-equivalent check
should be confirmed against `dotcl/dotcl`'s current `Runtime.DotNet.cs` /
`Runtime.Type.cs` — this doc doesn't assert one exists verbatim by that
name, only that the underlying CLR call is straightforward to expose.)

## Suggested regression test

Any class with two overloads sharing a common parameter prefix, where:

- one overload is fully required at arity N, and
- another overload is arity N (or more) but has a *usable default* on a
  trailing parameter making its effective minimum arity ≤ N,

and where the parameter(s) at the position(s) distinguishing the two
overloads are **not** one of `System.Double`/`Single`/`Int32`/`Int64`/
`Int16`/`Byte`/`Decimal`/`Boolean`/`String` (i.e. exercise the weak
fallback branch) — e.g. exactly the `MediaPlayer.Play(Song)` /
`Play(SongCollection, Int32 = 0)` pair above — should be added as a fixture
and asserted to:

1. Produce the correct overload-selection ordering when called with the
   minimum number of arguments (should hit the fully-required, no-default
   overload, not the effective-arity-tied default overload).
2. Correctly distinguish the two overloads when called with an argument of
   each overload's actual respective type, once issue #2 above is also
   fixed.

## Where this was found / local workaround applied

Downstream repo: `dotcl-dungeonslime`, file `audio-controller.lisp`,
method `play-song`. The generated `media-player:play` wrapper was bypassed
at the call site with a direct `(dotnet:static "Microsoft.Xna.Framework.Media.MediaPlayer" "Play" song)`
call, since hand-editing generated `cspackages/*.lisp` files is against
that project's conventions (they're regenerated via `make cspackages`).
This workaround should be reverted once this generator bug is fixed and
`cspackages/` is regenerated — at which point `(media-player:play song)`
should be restored in `audio-controller.lisp`.
